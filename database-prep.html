01Indexing — the most asked topic
CoreWhat is an index and how does it work?▶

An index is a data structure (usually a B-tree) that speeds up data retrieval, like the index at the back of a book. Instead of scanning every row (a table scan), the database can jump straight to the data.

The tradeoff: indexes speed up reads but slow down writes (inserts/updates/deletes must also update the index) and take extra storage. So you index columns you frequently search, filter, or join on — not everything.

MidClustered vs non-clustered index — what's the difference?▶
  • Clustered index — determines the physical order of data in the table. There can be only ONE per table (usually the primary key). The data IS the index — leaf nodes contain the actual rows.
  • Non-clustered index — a separate structure that points to the data. You can have many. Leaf nodes contain pointers to the actual rows.

Analogy: a clustered index is like a phone book sorted by last name — the data itself is ordered. A non-clustered index is like the index at the back of a textbook — a separate list pointing to page numbers.

Your experience
"At Mainframe I added composite indexes on the columns we filtered by most — like UserId and CreatedDate together — which dramatically cut query time on transaction tables with millions of rows."
MidWhat is a composite index and does column order matter?▶

A composite index covers multiple columns. Column order matters a lot — this is the "leftmost prefix" rule.

An index on (UserId, CreatedDate) can be used for queries filtering on UserId alone, or UserId + CreatedDate. But it CANNOT efficiently serve a query filtering only on CreatedDate — because CreatedDate isn't the leading column.

-- Index on (UserId, CreatedDate)
CREATE INDEX IX_Bets_User_Date ON Bets (UserId, CreatedDate);

-- ✓ Uses the index (UserId is leading column)
SELECT * FROM Bets WHERE UserId = 42;

-- ✓ Uses the index fully
SELECT * FROM Bets WHERE UserId = 42 AND CreatedDate > '2024-01-01';

-- ✗ Does NOT use index efficiently (CreatedDate not leading)
SELECT * FROM Bets WHERE CreatedDate > '2024-01-01';
Senior-level point
Put the most selective column (the one that filters out the most rows) or the most frequently queried column first. Getting composite index order right is exactly the kind of optimisation you did at Mainframe — lead with that.
SeniorWhat is a covering index?▶

A covering index includes all the columns a query needs — so the database can answer the query entirely from the index without touching the table at all. In SQL Server you add extra columns with the INCLUDE clause.

-- Index covers the query completely
CREATE INDEX IX_Bets_User ON Bets (UserId)
    INCLUDE (Amount, Status);

-- This query is fully served by the index — no table lookup
SELECT Amount, Status FROM Bets WHERE UserId = 42;
02Transactions & ACID
CoreWhat does ACID stand for?▶
  • Atomicity — all operations in a transaction succeed, or none do. No partial completion.
  • Consistency — a transaction brings the database from one valid state to another, respecting all rules and constraints.
  • Isolation — concurrent transactions don't interfere with each other.
  • Durability — once committed, the data survives even a crash or power loss.
Perfect example from your work
"In our betting platform, placing a bet and deducting the balance is one atomic transaction. Atomicity guarantees you can't deduct the balance without recording the bet, or vice versa — both happen or neither does."
SeniorExplain transaction isolation levels▶

Isolation levels control how much one transaction sees of another's uncommitted changes. From least to most strict:

  • Read Uncommitted — can read uncommitted changes (dirty reads). Fastest, least safe.
  • Read Committed — only reads committed data. SQL Server default. Prevents dirty reads.
  • Repeatable Read — guarantees rows read won't change during the transaction. Prevents non-repeatable reads.
  • Serializable — full isolation, transactions behave as if run one at a time. Safest, slowest.
The three read phenomena to know
  • Dirty read — reading uncommitted data that might be rolled back
  • Non-repeatable read — same query returns different values within a transaction
  • Phantom read — new rows appear in repeated queries within a transaction
Senior insight
Higher isolation = more safety but more locking = less concurrency. It's a tradeoff. For a betting balance you might need a higher level to prevent double-spending, but for reading odds, Read Committed is fine. Knowing this tradeoff signals senior-level understanding.
SeniorWhat is a deadlock and how do you handle it?▶

A deadlock happens when two transactions each hold a lock the other needs — both wait forever. e.g. Transaction A locks row 1 and wants row 2; Transaction B locks row 2 and wants row 1.

How databases handle it: SQL Server detects deadlocks and kills one transaction (the "victim"), rolling it back so the other can proceed.

How you prevent it: access resources in a consistent order across all transactions, keep transactions short, and use appropriate isolation levels. In code, retry the victim transaction.

SeniorOptimistic vs pessimistic locking▶
  • Pessimistic locking — lock the row when you read it, blocking others until you're done. Safe but reduces concurrency. Use when conflicts are likely.
  • Optimistic locking — don't lock; instead check at write time whether the data changed since you read it (using a version/timestamp column). If it changed, reject and retry. Better concurrency. Use when conflicts are rare.
Relevant to betting
"For balance deductions where double-spending must be prevented, you have to think carefully about locking strategy — optimistic locking with a version check works well when conflicts are rare, and avoids the concurrency cost of locking every read."
03Schema design & normalization
MidWhat is normalization? Explain the normal forms.▶

Normalization organises tables to reduce redundancy and improve data integrity. The main forms:

  • 1NF — each column holds atomic (indivisible) values; no repeating groups
  • 2NF — 1NF + every non-key column depends on the WHOLE primary key (no partial dependency)
  • 3NF — 2NF + no non-key column depends on another non-key column (no transitive dependency)

Most well-designed databases aim for 3NF.

SeniorWhen would you denormalize a database?▶

Denormalization deliberately introduces redundancy to improve read performance — trading storage and write complexity for faster reads.

When: read-heavy systems where joins are expensive, reporting/analytics tables, or when a frequently-run query joins many tables. e.g. storing a precomputed "total bets" count rather than counting every time.

The tradeoff: you now have to keep the redundant data in sync on writes. Only denormalize when you have a proven performance need — premature denormalization creates bugs.

MidWhat is the N+1 query problem?▶

The N+1 problem: you run 1 query to get a list of N items, then N more queries to get related data for each item — N+1 total queries instead of a smart 1 or 2.

Classic in ORMs like EF Core when lazy loading. Fix it with eager loading (.Include()) to fetch related data in one query.

// ❌ N+1 problem — 1 query for bets, then 1 per bet for user
var bets = await _db.Bets.ToListAsync();
foreach (var bet in bets)
    Console.WriteLine(bet.User.Name);  // triggers a query each time!

// ✓ Fixed — one query with a join
var bets = await _db.Bets.Include(b => b.User).ToListAsync();
04Query optimization & practical SQL
MidHow do you find and fix a slow query?▶
  • Read the execution plan — SQL Server shows how it runs the query. Look for table scans (bad) vs index seeks (good)
  • Add missing indexes — on WHERE, JOIN, ORDER BY columns
  • Avoid SELECT * — fetch only needed columns
  • Avoid functions on indexed columns — WHERE YEAR(date) = 2024 can't use an index; use a range instead
  • Check for N+1 — batch queries where possible
  • Use pagination — never return unbounded result sets
Your direct experience
"This is exactly what I did at Mainframe — I used execution plans to identify table scans, added composite indexes on the filtered columns, and switched from selecting full entities to projecting only needed fields. Query times dropped significantly on our largest transaction tables."
MidExplain the difference between the JOIN types▶
  • INNER JOIN — only rows with matches in both tables
  • LEFT JOIN — all rows from the left table, matched rows from the right (NULLs where no match)
  • RIGHT JOIN — all rows from the right, matched from the left
  • FULL OUTER JOIN — all rows from both, matched where possible
SeniorWHERE vs HAVING — what's the difference?▶

WHERE filters rows BEFORE grouping. HAVING filters groups AFTER a GROUP BY aggregation.

-- WHERE filters individual rows first, HAVING filters the groups
SELECT UserId, COUNT(*) AS BetCount
FROM Bets
WHERE Amount > 100          -- filter rows before grouping
GROUP BY UserId
HAVING COUNT(*) > 5;        -- filter groups after aggregation
