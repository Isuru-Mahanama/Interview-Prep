01 .NET Core — Interview questions
Fundamentals — every interview asks these
Basic What is the difference between .NET Framework and .NET Core? ▶
Answer

.NET Framework is Windows-only, older, and ships with Windows. It cannot run on Linux or Mac.

.NET Core (now just called .NET from version 5+) is cross-platform, open source, faster, and designed for modern cloud and microservice architectures. It runs on Windows, Linux, and Mac.

At Mainframe you use .NET Core — which is the right choice for any new production system today.

How to say it in an interview
"In my current role at Mainframe we use .NET Core specifically because of its performance advantages and cross-platform support — it runs identically on our development machines and Linux production servers."
Basic What is middleware in ASP.NET Core and how does it work? ▶
Answer

Middleware is software assembled into the request pipeline to handle requests and responses. Each middleware component can: process the request, pass it to the next middleware, or short-circuit the pipeline.

They run in the order they are added in Program.cs — order matters.

// Order matters — authentication before authorization
app.UseAuthentication();
app.UseAuthorization();
app.UseMiddleware<LoggingMiddleware>();
app.MapControllers();

Common built-in middleware: UseRouting, UseAuthentication, UseAuthorization, UseStaticFiles, UseCors, UseExceptionHandler.

Senior angle
Mention that at Mainframe you used middleware for cross-cutting concerns like logging and authentication — it keeps that logic out of your controllers and centralised in one place.
Basic What is dependency injection and why is it important? ▶
Answer

Dependency Injection (DI) is a design pattern where a class receives its dependencies from outside rather than creating them itself. ASP.NET Core has DI built in.

Why it matters: loose coupling, easier testing, and easier to swap implementations.

// Register in Program.cs
builder.Services.AddScoped<IPaymentService, IPayService>();

// Inject into controller — no "new" keyword needed
public class PaymentController : ControllerBase
{
    private readonly IPaymentService _paymentService;

    public PaymentController(IPaymentService paymentService)
    {
        _paymentService = paymentService;
    }
}
Three lifetimes — must know all three
  • Transient — new instance every time it's requested. Use for lightweight, stateless services.
  • Scoped — one instance per HTTP request. Most common — use for database contexts.
  • Singleton — one instance for the entire application lifetime. Use for caching, configuration.
Common interview trap
Interviewers often ask "what happens if you inject a Scoped service into a Singleton?" — the answer is a captive dependency problem. The Singleton holds a reference to a Scoped service that should have been disposed, causing bugs. Always inject services of equal or longer lifetime.
Async & Performance
Mid What is async/await and when should you use it? ▶
Answer

async/await is C#'s way of writing asynchronous code that looks synchronous. When you await an operation, the thread is released to handle other requests — it doesn't sit there blocked waiting.

Use it for I/O-bound operations: database calls, HTTP requests, file reads. Don't use it for CPU-bound work — use Task.Run() for that instead.

// ❌ Synchronous — blocks the thread
public BetSlip GetBetSlip(int id)
{
    return _db.BetSlips.Find(id);
}

// ✓ Asynchronous — releases thread while waiting
public async Task<BetSlip> GetBetSlipAsync(int id)
{
    return await _db.BetSlips.FindAsync(id);
}
Real world connection
"In our betting platform at Mainframe, all our database and third-party API calls are async — because under load, blocking threads would exhaust the thread pool and kill throughput."
Mid What is the difference between IEnumerable, IQueryable, and IList? ▶
Answer
  • IEnumerable — in-memory iteration. Query executes immediately when enumerated. Best for in-memory collections.
  • IQueryable — deferred execution. Query is built up and sent to the database as SQL only when enumerated. Best for database queries — lets EF Core optimise the SQL.
  • IList — in-memory, supports indexing and modification. Use when you need to add/remove items.
// IQueryable — filter happens IN the database (efficient)
IQueryable<Bet> query = _db.Bets.Where(b => b.UserId == userId);

// IEnumerable — ALL records loaded first, THEN filtered in memory (slow!)
IEnumerable<Bet> bets = _db.Bets.AsEnumerable().Where(b => b.UserId == userId);
Senior-level point to make
Using IEnumerable on a large table is a common performance bug — it loads every row into memory before filtering. Always use IQueryable for database queries and only materialise with .ToList() or .ToListAsync() when you're done composing the query.
Web API & REST
Mid What are the HTTP status codes and when do you use each? ▶
Answer — must know these cold
  • 200 OK — success, returns data
  • 201 Created — resource created successfully (POST)
  • 204 No Content — success but nothing to return (DELETE)
  • 400 Bad Request — client sent invalid data
  • 401 Unauthorized — not authenticated (no token)
  • 403 Forbidden — authenticated but not allowed
  • 404 Not Found — resource doesn't exist
  • 409 Conflict — duplicate or state conflict
  • 500 Internal Server Error — something broke on the server
Interview tip
Interviewers ask "what's the difference between 401 and 403?" very often. 401 = you haven't logged in. 403 = you're logged in but don't have permission. Simple but many people get it wrong.
Mid How does JWT authentication work in ASP.NET Core? ▶
Answer

JWT (JSON Web Token) has three parts: Header.Payload.Signature — all base64 encoded and dot-separated.

The flow: user logs in → server validates credentials → server creates a JWT signed with a secret key → client stores the token → client sends it in every request as Authorization: Bearer <token> → server validates the signature and reads the claims.

// Setup in Program.cs
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
            IssuerSigningKey = new SymmetricSecurityKey(
                Encoding.UTF8.GetBytes(config["Jwt:Secret"]))
        };
    });

// Protect an endpoint
[Authorize]
[HttpGet("profile")]
public IActionResult GetProfile() => Ok(_userService.GetCurrentUser());
Real world connection
You implemented JWT in HirMe for your freelancing platform — mention that. Also mention you handled authentication in the betting platform's third-party API integrations at Mainframe.
Senior-level questions
Senior How do you handle database transactions in .NET Core to ensure data consistency? ▶
Answer

In EF Core, SaveChanges() wraps everything in a transaction by default. For complex operations spanning multiple steps, use explicit transactions:

using var transaction = await _db.Database.BeginTransactionAsync();
try
{
    // Place bet
    _db.Bets.Add(bet);
    await _db.SaveChangesAsync();

    // Deduct balance
    user.Balance -= bet.Amount;
    await _db.SaveChangesAsync();

    await transaction.CommitAsync();
}
catch
{
    await transaction.RollbackAsync(); // both operations undone
    throw;
}

This is critical in a betting system — if the bet is placed but the balance deduction fails, you have a consistency problem. The transaction ensures both succeed or both are rolled back.

Real scenario from your experience
"In our betting platform at Mainframe, every bet placement is wrapped in a transaction — we place the bet AND deduct the user's balance atomically. If either step fails, the whole operation rolls back. A partial failure would mean a user placed a bet without paying for it."
Senior What is the repository pattern and when would you use it? ▶
Answer

The Repository pattern abstracts the data access layer behind an interface, so your business logic doesn't depend directly on EF Core or any specific database technology.

// Interface — business logic depends on this abstraction
public interface IBetRepository
{
    Task<Bet> GetByIdAsync(int id);
    Task AddAsync(Bet bet);
    Task<IEnumerable<Bet>> GetByUserAsync(int userId);
}

// Implementation — the actual EF Core code
public class BetRepository : IBetRepository
{
    private readonly AppDbContext _db;
    public BetRepository(AppDbContext db) => _db = db;

    public async Task<Bet> GetByIdAsync(int id)
        => await _db.Bets.FindAsync(id);
}

Benefits: easier unit testing (mock the interface), swap database without touching business logic, cleaner separation of concerns.

When NOT to use it: simple CRUD applications where the abstraction adds complexity without benefit. EF Core's DbContext is already an abstraction.

Senior How would you optimise a slow SQL query in a high-traffic system? ▶
Answer — give this as a structured process
  • 1. Identify — use SQL Server's execution plan or EF Core logging to find the slow query
  • 2. Indexes — add indexes on columns used in WHERE, JOIN, and ORDER BY. Composite indexes for multi-column filters
  • 3. Avoid N+1 — use .Include() in EF Core to eager-load related data instead of making one query per record
  • 4. Select only what you need — use .Select() to fetch specific columns, not entire entities
  • 5. Pagination — never return all rows. Use .Skip().Take() for paging
  • 6. Caching — cache frequently read, rarely changed data in memory
  • 7. Raw SQL — for complex queries, drop down to raw SQL or stored procedures
Real scenario from your experience
"At Mainframe I worked on optimising queries against transaction tables with millions of rows. The main wins were adding composite indexes on UserId + CreatedDate columns we were filtering on, and switching from loading full entities to projecting only the fields needed for the response."
Design Patterns & SOLID
Mid What are the SOLID principles? ▶
Answer — memorise all five
  • S — Single Responsibility — a class should have one reason to change. One job.
  • O — Open/Closed — open for extension, closed for modification. Add behaviour without changing existing code.
  • L — Liskov Substitution — subtypes must be substitutable for their base types without breaking things.
  • I — Interface Segregation — many small specific interfaces are better than one large general one.
  • D — Dependency Inversion — depend on abstractions (interfaces), not concrete implementations. This is what DI enables.
Interview tip
Don't just recite them — give a one-line example of each. For Dependency Inversion, mention that ASP.NET Core's built-in DI is a direct application of this principle, which ties back to your daily work.
Mid Explain the Singleton, Factory, and Strategy patterns ▶
Answer — the three most-asked patterns
  • Singleton — ensures only one instance of a class exists. Use for shared resources like configuration or a logger. In .NET Core, register a service as Singleton in DI.
  • Factory — creates objects without exposing the creation logic. The caller asks the factory for an object and gets the right type back. Useful when object creation is complex or depends on conditions.
  • Strategy — defines a family of interchangeable algorithms. Swap behaviour at runtime. e.g. different payment processing strategies (card vs IPay) behind a common interface.
// Strategy pattern — swap payment methods behind an interface
public interface IPaymentStrategy
{
    Task<PaymentResult> ProcessAsync(decimal amount);
}

public class IPayStrategy : IPaymentStrategy { /* IPay logic */ }
public class CardStrategy : IPaymentStrategy { /* card logic */ }

// The context picks a strategy at runtime
public class PaymentProcessor
{
    private readonly IPaymentStrategy _strategy;
    public PaymentProcessor(IPaymentStrategy strategy) => _strategy = strategy;
}
Connect to your experience
"At Mainframe I integrated IPay as a payment gateway. A clean way to structure multiple payment providers is the Strategy pattern — each provider implements a common interface, so adding a new one doesn't touch existing code. That's the Open/Closed principle in action."
Senior What is the Unit of Work pattern and how does it relate to Repository? ▶
Answer

The Unit of Work pattern tracks changes across multiple repositories and commits them together as a single transaction. Repository handles one entity type; Unit of Work coordinates saving across many.

Key insight: EF Core's DbContext already implements both patterns — it IS a Unit of Work (SaveChanges commits everything) and DbSet acts as a repository. So adding your own layer on top is often redundant unless you need to abstract away EF Core entirely.

Senior-level nuance
A great senior answer mentions that wrapping EF Core in custom Repository + Unit of Work is often over-engineering, since DbContext already provides both. Knowing when NOT to apply a pattern is as valuable as knowing the pattern.
01 Angular — Interview questions
Fundamentals — every interview asks these
Basic What is Angular and how is it different from React? ▶
Answer

Angular is a full framework — it comes with everything built in: routing, forms, HTTP client, DI, state management patterns. It uses TypeScript by default and follows an opinionated structure.

React is a UI library — it only handles the view layer. You pick your own router, state management, and HTTP library.

Key Angular concepts: Components, Modules, Services, Directives, Pipes, Dependency Injection, RxJS Observables.

How to say it
"I use Angular daily at Mainframe for our betting platform frontend. I chose it because its opinionated structure scales well in large teams — everyone follows the same patterns, which reduces cognitive overhead."
Basic What is the difference between a Component and a Service in Angular? ▶
Answer

Component — controls a piece of the UI. Has a template (HTML), styles (CSS), and logic (TypeScript). Responsible for displaying data and handling user interactions.

Service — handles business logic, data fetching, and shared state. Has no UI. Injected into components via DI. Keeps components thin and focused on the view.

// Service — handles data, no UI
@Injectable({ providedIn: 'root' })
export class BetService {
  constructor(private http: HttpClient) {}

  placeBet(bet: Bet): Observable<BetResult> {
    return this.http.post<BetResult>('/api/bets', bet);
  }
}

// Component — handles UI, delegates logic to service
@Component({ selector: 'app-bet-slip', templateUrl: './bet-slip.component.html' })
export class BetSlipComponent {
  constructor(private betService: BetService) {}

  onSubmit() {
    this.betService.placeBet(this.bet).subscribe(result => this.handleResult(result));
  }
}
Basic What are Angular directives? What is the difference between structural and attribute directives? ▶
Answer

Directives extend HTML with new behaviour.

Structural directives — change the DOM structure by adding or removing elements. Prefixed with *.

  • *ngIf — conditionally adds/removes an element
  • *ngFor — repeats an element for each item in a list
  • *ngSwitch — switches between views

Attribute directives — change the appearance or behaviour of an existing element without changing the DOM structure.

  • ngClass — adds/removes CSS classes
  • ngStyle — applies inline styles
  • ngModel — two-way data binding
RxJS & Observables — very common at senior level
Mid What is an Observable and how is it different from a Promise? ▶
Answer
  • Promise — handles one future value. Executes immediately when created. Cannot be cancelled. Used in plain JavaScript.
  • Observable — handles zero or more values over time. Lazy — only executes when subscribed. Can be cancelled. Can be transformed with operators like map, filter, debounceTime.
// Promise — one value, executes immediately
const promise = fetch('/api/bets').then(r => r.json());

// Observable — lazy, cancellable, can emit multiple values
const bets$ = this.http.get<Bet[]>('/api/bets').pipe(
  map(bets => bets.filter(b => b.status === 'active')),
  catchError(err => { console.error(err); return of([]); })
);

// Subscribe to get the value
const sub = bets$.subscribe(bets => this.bets = bets);

// Cancel when done (prevents memory leaks)
sub.unsubscribe();
Memory leak warning — interviewers love this
Always unsubscribe from Observables in ngOnDestroy, or use the async pipe in the template which unsubscribes automatically. Forgetting to unsubscribe is the most common Angular memory leak.
Mid What is the difference between Subject, BehaviorSubject, and ReplaySubject? ▶
Answer
  • Subject — emits values to current subscribers only. Late subscribers miss previous values.
  • BehaviorSubject — stores the last value. New subscribers immediately get the current value. Most commonly used for shared state.
  • ReplaySubject(n) — replays the last n values to new subscribers.
// BehaviorSubject — most common for state management
private currentUser$ = new BehaviorSubject<User | null>(null);

// Update state
setUser(user: User) { this.currentUser$.next(user); }

// Read state anywhere
getUser() { return this.currentUser$.asObservable(); }
Interview tip
BehaviorSubject is the go-to for sharing state between components in Angular without a full state management library like NgRx. Mention you use it in your services to share user session data or application state.
Performance & Senior-level
Mid What is Change Detection in Angular and how does OnPush work? ▶
Answer

Angular's change detection checks if the UI needs to be updated. By default (Default strategy), Angular checks every component in the tree on every event — clicks, HTTP responses, timers.

OnPush strategy — Angular only checks a component when its @Input references change, or an Observable it subscribes to emits. This dramatically reduces unnecessary checks in large applications.

@Component({
  selector: 'app-bet-list',
  templateUrl: './bet-list.component.html',
  changeDetection: ChangeDetectionStrategy.OnPush  // ← add this
})
export class BetListComponent {
  @Input() bets: Bet[] = [];  // only re-renders when this reference changes
}
Senior angle
"In our betting platform, real-time odds updates were causing the entire component tree to re-render constantly. Switching list components to OnPush and using the async pipe reduced unnecessary renders significantly."
Senior How do you communicate between components in Angular? ▶
Answer — four ways, know all of them
  • @Input / @Output — parent to child (@Input) and child to parent (@Output with EventEmitter). Use for direct parent-child relationships.
  • Service with BehaviorSubject — shared service holds state, components subscribe. Use for sibling components or distantly related components.
  • ViewChild — parent accesses child component directly. Use sparingly.
  • NgRx / State management — centralised store for complex applications. Use when service-based state becomes hard to manage.
// Parent → Child with @Input
<app-bet-slip [bet]="selectedBet"></app-bet-slip>

// Child → Parent with @Output
@Output() betPlaced = new EventEmitter<BetResult>();
onSubmit() { this.betPlaced.emit(result); }
Senior What is lazy loading in Angular and why does it matter? ▶
Answer

Lazy loading loads feature modules only when the user navigates to that route — not upfront. This reduces the initial bundle size, making the app load faster.

// Without lazy loading — everything loads upfront
{ path: 'admin', component: AdminComponent }

// With lazy loading — AdminModule only loads when user visits /admin
{
  path: 'admin',
  loadChildren: () => import('./admin/admin.module')
    .then(m => m.AdminModule)
}

Combined with route guards, lazy loading also means users who don't have access to a section never even download that code.

Real scenario from your experience
"In the betting platform, the admin reporting module is lazily loaded — regular users never download it. This kept the initial bundle small and improved load time for the majority of users who only use the betting interface."
