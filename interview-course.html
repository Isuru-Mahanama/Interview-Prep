02 How to behave in a coding interview

Before you learn any topic, learn this. This is what separates candidates who get offers from those who don't — it's not just about solving the problem.

1
Clarify before you touch the keyboard
Ask questions. Restate the problem in your own words. Confirm input types, edge cases, and constraints. Interviewers deliberately leave things vague to see if you ask.
"Just to confirm — the input array can have duplicates, and I should return indices, not values, right?"
2
Think out loud — always
The interviewer cannot see inside your head. Narrate every thought. A candidate who explains their thinking and gets a wrong answer is better than one who stays silent and gets the right answer.
"I'm thinking of brute force first — nested loops, O(n²). That works but let me see if I can do better with a hash map..."
3
Start with brute force, then optimise
Never jump straight to the optimal solution. Say the brute force first, explain why it's slow, then work toward something better. This shows structured thinking.
"Brute force is O(n²) with two nested loops. I can bring it to O(n) using a hash map to store values I've seen."
4
State time and space complexity
After writing your solution, always say the Big O complexity unprompted. This is expected at senior level — not saying it is a red flag.
"This runs in O(n) time and O(n) space because of the hash map."
5
Test your own code with examples
Once you've written the code, walk through it manually with a simple input. Then test edge cases — empty array, single element, all duplicates, negative numbers.
"Let me trace through with input [2,7,11,15], target=9... index 0, value 2, I need 7, check hash map... found it at index 1. Returns [0,1]. Correct."
6
Never freeze in silence
If you're stuck, say so out loud. Ask for a hint if needed — that's allowed. "I'm stuck on how to handle the duplicate case, could I get a nudge?" is far better than 3 minutes of silence.

01
Arrays & Strings
Week 1 · Foundation of every interview
Start here ▶
What you need to know

Arrays are the most common data structure in interviews. You must be fluent in: iterating, indexing, slicing, reversing, sorting, and searching. For strings — the key insight is that a string is just an array of characters.

The most important pattern: Two Pointers. Use two indices (start and end, or slow and fast) to solve problems in O(n) that would otherwise take O(n²).

Two pointers pattern — example
# Check if a string is a palindrome
def is_palindrome(s):
    left, right = 0, len(s) - 1
    
    while left < right:
        if s[left] != s[right]:
            return False
        left += 1
        right -= 1
    
    return True

# Time: O(n)  Space: O(1)
Sliding window pattern — example
# Maximum sum subarray of size k
def max_sum_subarray(arr, k):
    window_sum = sum(arr[:k])
    max_sum = window_sum
    
    for i in range(k, len(arr)):
        window_sum += arr[i] - arr[i - k]  # slide
        max_sum = max(max_sum, window_sum)
    
    return max_sum

# Time: O(n)  Space: O(1)
Interview tip
When you see a problem involving a contiguous subarray or substring — your first thought should be sliding window. When you see a sorted array — your first thought should be two pointers.

02
Hash Maps & Sets
Week 1 · Optimises almost every brute force
Week 1 ▶
Why hash maps matter

A hash map gives you O(1) average lookup, insert, and delete. The single most common optimisation in interviews is: "instead of searching the array again (O(n)), store what you've seen in a hash map (O(1) lookup)".

If your brute force uses two nested loops, a hash map will almost always bring it from O(n²) to O(n).

The classic pattern
# Two Sum — hash map solution
def two_sum(nums, target):
    seen = {}  # value -> index
    
    for i, num in enumerate(nums):
        complement = target - num
        
        if complement in seen:
            return [seen[complement], i]
        
        seen[num] = i
    
    return []

# Time: O(n)  Space: O(n)
# vs brute force: O(n²) time, O(1) space

03
Linked Lists
Week 2 · Pointer manipulation
Week 2 ▶
What you need to know

Linked list problems are about pointer manipulation. The key patterns are: fast & slow pointers (detect cycles, find middle), reversing a list, and merging lists.

Always draw it out on paper first. Pointer bugs are hard to spot in your head but obvious on paper.

Fast & slow pointer — detect cycle
def has_cycle(head):
    slow, fast = head, head
    
    while fast and fast.next:
        slow = slow.next        # moves 1 step
        fast = fast.next.next   # moves 2 steps
        
        if slow == fast:
            return True
    
    return False

# Time: O(n)  Space: O(1)
Reversing a linked list
def reverse_list(head):
    prev = None
    curr = head
    
    while curr:
        next_node = curr.next  # save next
        curr.next = prev       # reverse pointer
        prev = curr            # move prev forward
        curr = next_node       # move curr forward
    
    return prev

# Time: O(n)  Space: O(1)
Interview tip
Always check for null/None before accessing .next. Forgetting null checks is the most common linked list mistake. Say "I'll add null checks here" even if the interviewer hasn't asked.

04
Stacks & Queues
Week 2 · LIFO and FIFO patterns
Week 2 ▶
When to use each

Stack (LIFO) — use when you need to process things in reverse order, track opening/closing pairs (brackets), or undo operations. Implemented with a list in Python.

Queue (FIFO) — use for level-order traversal of trees, BFS in graphs, or processing items in arrival order. Use collections.deque in Python for O(1) operations.

Classic stack problem — valid parentheses
def is_valid(s):
    stack = []
    pairs = {')': '(', '}': '{', ']': '['}
    
    for char in s:
        if char in pairs:
            if not stack or stack[-1] != pairs[char]:
                return False
            stack.pop()
        else:
            stack.append(char)
    
    return not stack

# Time: O(n)  Space: O(n)

05
Binary Search
Week 3 · Eliminate half the search space each step
Week 3 ▶
The key insight

Binary search isn't just for finding a value in a sorted array. The real pattern is: any time you can eliminate half the search space based on a condition, you can use binary search.

Watch for phrases like "sorted array", "find the minimum", "find the boundary", "search in rotated array" — these all hint at binary search.

Template — memorise this
def binary_search(nums, target):
    left, right = 0, len(nums) - 1
    
    while left <= right:
        mid = left + (right - left) // 2  # avoids overflow
        
        if nums[mid] == target:
            return mid
        elif nums[mid] < target:
            left = mid + 1   # search right half
        else:
            right = mid - 1  # search left half
    
    return -1  # not found

# Time: O(log n)  Space: O(1)
Common mistake
Use mid = left + (right - left) // 2 not mid = (left + right) // 2. The second can cause integer overflow in languages like Java and C#. Always use the safe version — interviewers notice this.

Trees & Recursion
Week 4–5 · Very common at all levels
Week 4 ▶
The three traversals — must know all three
# Inorder: Left -> Root -> Right (gives sorted order for BST)
def inorder(root):
    if not root: return []
    return inorder(root.left) + [root.val] + inorder(root.right)

# Preorder: Root -> Left -> Right (copy a tree)
def preorder(root):
    if not root: return []
    return [root.val] + preorder(root.left) + preorder(root.right)

# Postorder: Left -> Right -> Root (delete a tree)
def postorder(root):
    if not root: return []
    return postorder(root.left) + postorder(root.right) + [root.val]

# All: Time O(n)  Space O(h) where h = height of tree
BFS — level order traversal
from collections import deque

def level_order(root):
    if not root: return []
    result, queue = [], deque([root])
    
    while queue:
        level = []
        for _ in range(len(queue)):
            node = queue.popleft()
            level.append(node.val)
            if node.left: queue.append(node.left)
            if node.right: queue.append(node.right)
        result.append(level)
    
    return result

# Time: O(n)  Space: O(n)

Graphs & BFS / DFS
Week 5–6 · Senior-level favourite
Week 5 ▶
When to use BFS vs DFS

BFS — use when you need the shortest path, or need to explore level by level. Uses a queue.

DFS — use when you need to explore all paths, detect cycles, or find connected components. Uses recursion or a stack.

DFS template
def dfs(graph, node, visited):
    if node in visited:
        return
    
    visited.add(node)
    # process node here
    
    for neighbour in graph[node]:
        dfs(graph, neighbour, visited)

# Time: O(V + E)  Space: O(V)
# V = vertices, E = edges

Dynamic Programming
Week 7 · Hardest topic — most rewarding to crack
Week 7 ▶
What DP actually is

Dynamic Programming is just recursion + memoization. If a recursive solution recalculates the same subproblems, store the results in a table and reuse them.

The two signs you need DP: (1) overlapping subproblems — the same calculation is done multiple times, and (2) optimal substructure — the best solution is built from best solutions to smaller problems.

Classic — Fibonacci with memoization
# Without DP: O(2ⁿ) — terrible
def fib_slow(n):
    if n <= 1: return n
    return fib_slow(n-1) + fib_slow(n-2)

# With memoization: O(n) — excellent
def fib(n, memo={}):
    if n in memo: return memo[n]
    if n <= 1: return n
    memo[n] = fib(n-1, memo) + fib(n-2, memo)
    return memo[n]

# Time: O(n)  Space: O(n)
