System Design Prep

System Design
interviews

The defining round for senior roles. There's no single right answer — interviewers watch how you think, handle tradeoffs, and scale a system. This guide gives you a repeatable framework plus the classic problems.

01The framework — use this every time

Never start drawing boxes immediately. Follow these steps in order — interviewers are specifically looking for this structured approach.

1Clarify requirements (5 min)
Ask about functional requirements (what it must do) and non-functional ones (scale, latency, availability). Never assume — the interviewer deliberately keeps it vague. Ask: "How many users? Read-heavy or write-heavy? What's the acceptable latency?"
2Estimate scale (5 min)
Do rough back-of-envelope math. Daily active users, requests per second, storage needed, bandwidth. This shows you think about scale from the start.
3Define the API (5 min)
Sketch the main endpoints. e.g. for a URL shortener: POST /shorten and GET /{shortCode}. This anchors the whole design.
4High-level design (10 min)
Draw the major components: client, load balancer, web servers, application servers, database, cache. Show how a request flows through them.
5Deep dive (10 min)
Pick the most interesting or critical component and go deep. Database schema, caching strategy, how you handle the bottleneck.
6Address bottlenecks & scale (10 min)
Identify single points of failure. Discuss database replication, sharding, caching, CDN, horizontal scaling. This is where senior candidates shine.
02Core concepts you must know
Scaling & availability
CoreHorizontal vs vertical scaling▶

Vertical scaling — add more power (CPU, RAM) to a single server. Simple but has a ceiling and a single point of failure.

Horizontal scaling — add more servers and distribute load. More complex but virtually unlimited and fault-tolerant. This is how big systems scale.

CoreWhat is a load balancer and why do you need one?▶

A load balancer distributes incoming requests across multiple servers so no single server is overwhelmed. It also provides failover — if one server dies, traffic routes to healthy ones.

Common algorithms: Round Robin (rotate evenly), Least Connections (send to least busy), IP Hash (same client → same server).

MidExplain caching and cache invalidation strategies▶

Caching stores frequently accessed data in fast memory (like Redis) to avoid hitting the database every time. Dramatically reduces latency and database load.

Strategies:

  • Cache-aside — app checks cache first, falls back to DB, then populates cache
  • Write-through — write to cache and DB simultaneously
  • Write-back — write to cache first, DB later (faster but risk of data loss)

Invalidation: TTL (time-to-live expiry), or explicit eviction when data changes.

Famous quote
"There are only two hard things in computer science: cache invalidation and naming things." Mention this — interviewers appreciate it and it shows you understand cache invalidation is genuinely hard.
Databases at scale
MidSQL vs NoSQL — when do you use each?▶
  • SQL (relational) — structured data, strong consistency, complex queries and joins, ACID transactions. Use for financial systems, anything needing consistency. Your betting platform uses SQL Server for exactly this reason.
  • NoSQL — flexible schema, horizontal scaling, high throughput. Types: document (MongoDB), key-value (Redis), column (Cassandra), graph (Neo4j). Use for massive scale, unstructured data, or when schema changes often.
Your real experience
"At Mainframe we use MS SQL Server for the betting platform specifically because financial transactions need ACID guarantees — you can't have eventual consistency when real money and bet settlements are involved."
SeniorWhat is database sharding and replication?▶

Replication — keep copies of the same data on multiple servers. Improves read performance (read from replicas) and availability (if one fails, others serve). Usually one primary for writes, multiple replicas for reads.

Sharding — split data across multiple databases by some key (e.g. user ID). Each shard holds a subset. Improves write performance and storage capacity but adds complexity — cross-shard queries are hard.

SeniorExplain the CAP theorem▶

CAP theorem says a distributed system can only guarantee two of three:

  • Consistency — every read gets the most recent write
  • Availability — every request gets a response
  • Partition tolerance — the system works despite network failures

Since network partitions are unavoidable in distributed systems, you're really choosing between consistency and availability. Banking systems choose consistency (CP). Social media feeds choose availability (AP) — a slightly stale like count is fine.

Communication patterns
MidWhat is a message queue and when do you use one?▶

A message queue (RabbitMQ, Kafka, Azure Service Bus) lets services communicate asynchronously. The producer puts a message on the queue; the consumer processes it later.

Why use it: decouples services, handles traffic spikes (buffer), enables retry on failure, and lets slow operations happen in the background.

Example: when a user places a bet, put "send confirmation email" on a queue instead of blocking the request while the email sends.

SeniorMonolith vs Microservices▶
  • Monolith — one deployable unit. Simpler to build, test, deploy initially. Gets harder to scale and maintain as it grows.
  • Microservices — many small independent services. Each scales and deploys independently, teams work autonomously. But adds huge operational complexity — networking, distributed transactions, monitoring.

Senior insight: Don't start with microservices. Start with a well-structured monolith and extract services only when you have a clear reason (a component that needs independent scaling, or a team that needs autonomy). Premature microservices is a classic mistake.

03Classic problems — practice these
PracticeDesign a URL shortener (like bit.ly)▶
Key points to hit
  • API: POST /shorten (long → short), GET /{code} (redirect)
  • Short code generation: base62 encoding of an auto-increment ID, or a hash. 7 characters of base62 = 3.5 trillion URLs
  • Database: key-value store mapping shortCode → longURL. NoSQL works well here
  • Scale reads: heavy caching (Redis) since it's read-heavy — most short URLs are clicked far more than created
  • Redirect: use 301 (permanent) or 302 (temporary — lets you track analytics)
PracticeDesign a rate limiter▶
Key points to hit
  • Purpose: prevent abuse, protect the system from being overwhelmed
  • Algorithms: Token Bucket (most common — refills tokens at a fixed rate), Sliding Window, Fixed Window Counter
  • Storage: Redis with TTL to track request counts per user/IP
  • Response: return HTTP 429 Too Many Requests when the limit is exceeded
  • Where: at the API gateway or load balancer level, before requests hit your servers
PracticeDesign a real-time betting/odds system (your domain!)▶
You have a genuine advantage here
  • Real-time odds: WebSockets or Server-Sent Events to push live odds to clients instead of polling
  • Bet placement: must be transactional and consistent (ACID) — deduct balance and record bet atomically
  • Handle concurrency: optimistic locking or DB transactions to prevent double-spending balance
  • Scale reads: cache odds heavily, they're read far more than written
  • Message queue: for settling bets and sending notifications asynchronously
  • Consistency over availability: for the money parts — you cannot have eventual consistency on account balances
